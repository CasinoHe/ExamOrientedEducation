# History Learning Framework

## Purpose of the History MVP

This document is the canonical pedagogy-facing reference for the first subject, History.

It defines how History should be learned, practiced, reviewed, and improved inside the product. It should guide later product and implementation decisions, but it is not a platform strategy document or an engineering module inventory.

## This File Covers

- History learning model
- review structure
- weak-point logic
- learning resources the product must support

## This File Does Not Cover

- platform and deployment strategy
- detailed engineering module breakdown
- implementation sequencing unless pedagogy requires it

## Why History Is a Good First Subject

History is a strong first subject because it supports modular learning, structured review, and exam-oriented training without requiring the system to solve many subject-specific symbolic problems.

It is a good fit because:

- History content can be organized into clear units, periods, and themes
- performance depends on both understanding and retention
- the subject works well with Mandarin-first source materials
- the learning loop can be improved through structure, recall, and targeted practice

## Core Learning Philosophy

The product should not function as a simple textbook viewer or summary library.

It should function as a History training system that helps students:

- learn core knowledge clearly
- connect facts into historical structure
- retain what they learned
- answer History questions more effectively
- correct recurring weaknesses efficiently

The goal is score improvement through a repeatable training loop.

## History Learning Loop

The core History learning loop should be:

1. learn
2. connect
3. memorize
4. answer
5. correct
6. review
7. repeat

This loop is stronger than chapter reading followed by occasional exercises because it continuously moves from understanding to recall to exam performance.

## Learning Course Structure

### Core Knowledge Learning

Students first need a clear understanding of the essential material in each unit.

This includes:

- historical events
- important figures
- timelines
- causes
- processes
- outcomes
- significance

The goal is to ensure the student can identify what happened and describe the key facts accurately.

### Structured Understanding

Students should not memorize isolated facts only. They also need to understand the relationships between them.

This layer should train students to see:

- why an event happened
- how one event influenced another
- what changed because of the event
- how similar events differ
- why an event matters in the wider historical picture

### Memory Reinforcement

History learning depends heavily on stable recall.

Students often lose points because they forget names, time order, significance, or links between events. The system therefore needs deliberate reinforcement of:

- people
- events
- terms
- timeline order
- historical relationships

### Question-Type Training

History knowledge must be converted into exam performance. The system should therefore train students through question forms rather than content review alone.

The product should support practice for:

- fact recognition
- timeline ordering
- cause-and-effect analysis
- comparison questions
- source-material interpretation
- short-answer or structured-response writing

### Weak-Point Review

When a student makes mistakes, review should target the exact weakness instead of sending the student back through a full chapter.

Weak-point review should focus on:

- one event
- one theme
- one confusion pair
- one causal misunderstanding
- one answer-structure problem

The goal is efficient correction with minimal wasted study time.

## Review Course Structure

### Daily Review

Daily review should be short, frequent, and focused on recent material.

It should reinforce:

- key terms
- important figures
- major events
- sequence and timeline

### Weak-Point Review

Weak-point review should be generated from mistakes and unstable recall.

It should emphasize:

- repeated errors
- weak concepts
- confusing event pairs
- unstable memory

### Cross-Unit Review

Students often strengthen History understanding by comparing across units instead of staying inside one chapter at a time.

Cross-unit review should help students connect:

- similar reforms
- different wars
- recurring social or political changes
- repeated historical patterns

### Exam-Oriented Review

Review should also prepare students for actual assessment conditions.

This layer should include:

- timed practice
- mixed-topic review sets
- answer-structure training
- question-type reinforcement

## Three Core Engines

### Knowledge Engine

The knowledge engine organizes what the student must know. It turns History source material into a structured knowledge system.

### Memory Engine

The memory engine helps the student retain knowledge over time. It reduces forgetting and strengthens recall.

### Exam Engine

The exam engine helps the student answer questions correctly and efficiently. It converts knowledge and memory into score improvement.

## Why These Engines Must Work Together

If one engine is missing, the system becomes weak:

- without the knowledge engine, the student lacks structure
- without the memory engine, the student forgets what was learned
- without the exam engine, the student knows content but cannot perform well in tests

The product should therefore be treated as a complete training system rather than a content library.

## History Knowledge Model

The internal learning structure for History should organize material around units that can be learned, reviewed, and assessed.

The primary elements are:

- period
- unit or chapter
- event
- figure or group
- cause
- process
- outcome
- significance
- comparison relationship
- question type

This structure should support both content organization and later weak-point diagnosis.

## Main Weak-Point Types in History

The first version should assume that History weaknesses often fall into a small number of patterns:

- factual recall weakness
- timeline confusion
- cause-and-effect confusion
- comparison weakness
- significance or explanation weakness
- answer-structure weakness

These weakness types should drive targeted review and practice design.

## Exam-Oriented Training Approach

The system should help students move from knowing content to using it in exam forms.

The training approach should:

- connect each learning unit to typical question forms
- train recognition, sequencing, explanation, comparison, and response structure
- reinforce both accuracy and clarity
- treat exam practice as part of the learning loop, not only as final testing

## Personalization Logic for History

History personalization should be based on mistakes, unstable recall, and repeated confusion patterns.

The system should:

- identify weak points from wrong answers and review performance
- distinguish between knowledge gaps and unstable memory
- assign targeted review instead of broad rereading
- revisit weak points until they become stable

## High-Value Learning Resources

The most valuable learning resources for History are the ones that improve both structure and recall.

These include:

- event summary cards
- timeline review
- comparison tables
- targeted drills
- mistake review items
- short-answer practice structures

These resources should be treated as essential learning assets, not optional extras.

## Design Principles for History Content

History content in the product should be:

- concise
- structured
- comparison-friendly
- memory-friendly
- exam-relevant
- easy to review in short study sessions

The student should be able to move quickly from reading to recalling to answering.

## Summary of the Teaching Model

The History MVP should support a learning system built around:

- a structured History knowledge model
- a repeatable loop of learn, connect, memorize, answer, correct, review, and repeat
- layered review including daily, weak-point, cross-unit, and exam-oriented review
- three cooperating engines: knowledge, memory, and exam

This teaching model is the pedagogical foundation for the product.
